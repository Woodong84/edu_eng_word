# 프롬프트/개발 이력 (edu_eng_word)

"영단어 마스터" 웹앱을 만들면서 오간 요청과 실제 반영 내용을 시간순으로 정리한 기록입니다. 웹앱 제작과 직접 관련된 항목만 남겼으며, 콘솔 설정 안내 등 상담성 내용은 말미의 부록에 압축했습니다. 현재 기능 사양은 [`SPEC.md`](./SPEC.md) 참고.

형식: **요청(요약) → 반영 내용 → 관련 PR**

---

## 1. 초기 구축 (업로드/구조화)

**요청**: 업로드한 `index.html`, `elementary-words.js` 두 파일을 분석해서 GitHub 저장소에 올리고, 추가 개발이 편하도록 디렉토리 구조를 잡아달라.

**반영**: 파일 분석 후 최초 커밋. 단일 HTML의 인라인 CSS/JS를 `css/style.css`, `js/app.js`로 분리하고 데이터를 `js/data/`로 이동, README 작성. (저장소가 비어 있어 최초 push가 그대로 main이 됨. 이후 모든 변경은 작업 브랜치 → PR → main 머지 흐름)

---

## 2. 프로필/폴더/레벨 테마/모바일 UI (PR #1)

**요청**: DB 없이 가능한 범위에서 — 로그인 관리(비밀번호 없는 프로필, 상위/하위 계정), 아이콘 선택, 레벨별 색/배경 변경, 레벨업 이펙트, 사용자별 단어 등록·폴더 관리, 등록 시 중복 검증. 모바일 우선, 기존 코드 리팩토링, 향후 클라우드 DB 고려.

**반영**: `storage.js`(저장 추상화)/`profiles.js`(계정) 분리, 프로필별 데이터 키 분리(`p:<id>:*`), 폴더 CRUD, 레벨 티어 테마 + 레벨업 파티클, 중복/형식 검증 리포트, 하단 고정 탭바 등 모바일 우선 UI, 레거시 데이터 자동 마이그레이션.

## 3. 모바일 소리(TTS) 문제 해결 (PR #2~#4)

**요청**: 모바일에서 다시 듣기가 무음 → 원인 규명·수정, 스피커 클릭 후 입력창 자동 포커스. 이후에도 무음 → 재진단. 카카오톡에서 열면 자동으로 크롬으로 열리게 해달라.

**반영**:
- PR #2: 원인은 모바일 크롬의 `cancel()` 직후 `speak()` 무시 버그 — 재생 중일 때만 cancel 후 80ms 지연 재생, iOS 제스처 컨텍스트 유지, utterance GC 방지. 스피커 탭 시 입력창 포커스
- PR #3: 잔여 원인은 카카오톡 인앱 브라우저의 TTS 미지원 — 안내 토스트 추가. 단어 카드 탭=발음 재생으로 변경(사전은 📖 버튼 분리)
- PR #4: `kakaotalk://web/openExternal` 스킴으로 외부 브라우저 자동 이동(안드로이드 기타 인앱은 크롬 intent), 실패 시에도 앱 사용 가능

## 5. 관리자 설정 분리 + 상태 탭 (PR #6, #7)

**요청**: 설정 메뉴는 프로필명 마스터/멋쟁이아빠만, 프로필 전환 왼쪽 배치. 오답 오른쪽에 [상태] 탭 — 레벨/학습 건수/오답 건수, 건수 클릭 시 목록. 학습한 단어 목록을 정답만/혼합/오답만 색으로 구분(오답만=기존 빨강 유지).

**반영**: 설정을 헤더 버튼으로 이동(관리자 전용+직접 진입 차단), 상태 탭 신설(요약 카드+상세 목록), 카드 색상 3종(초록/주황/빨강), 공용 카드 함수로 리팩토링.

## 6. 프로필 조회 범위/검색 + 문제 수 설정 (PR #8~#11)

**요청**: 하위=자신만, 상위=자신+하위, 관리자=전체 조회. 문제 수를 5개~범위 전체에서 선택(랜덤 출제). 범위 제한 때문에 관리자로 못 돌아감 → 이름 검색 추가, 탭바 간격. 검색 예시 문구에서 관리자 이름 제거. 문제 수 설정을 단어장(구 학습) 탭으로 이동+기본 5개, 퀴즈 빈칸에 "Hint" 라벨.

**반영**: viewer 기반 조회 범위 제한(최초 접속은 전체), 정확한 이름 일치 검색(부분 검색 불허), 문제 수 select(5단위, 프로필별 저장, 랜덤 출제), Hint 라벨, placeholder 예시를 "상상동물"로 교체, 캐시 무효화(`?v=N`) 도입.

## 7. Firebase 연동 — 로그인/마스터 DB/이력/랭킹 (PR #13)

**요청**: 무료 DB를 구성해 사용자 로그인 관리, Word 마스터 데이터 관리, 사용이력, 랭킹을 만들자(Firebase 선정). 랭킹은 주간 기준 + 월간은 주간 누적, 월 변경 시 리셋. 이력은 영구 보관. 스키마 설계 확인 후 진행.

**반영**: `firebase-init.js`/`cloud.js`/`firestore.rules` 신설. 익명 자동 로그인 + 관리자 이메일 로그인, `profiles`(통계 미러링)/`history`(문항별 로그)/`masterWords`(문서 ID=단어) 컬렉션, 주간·월간 lazy reset 랭킹, 프리셋 300개 시드 버튼. 로컬 우선 설계로 Firebase 불가 상태에서도 전 기능 동작(fire-and-forget + 5초 타임아웃).

## 부록 A. 상담/운영 항목 (코드 변경 없음, 요약)

- **GitHub Pages**: Settings → Pages → Deploy from a branch(main, /root)로 활성화. 실수로 Unpublish를 눌러도 Branch가 main으로 남아 있으면 게시 유지 상태
- **원격 세션**: Claude Code on the web은 클라우드 컨테이너 실행이라 사용자 PC 전원과 무관하게 작업 지속
- **DB 선정**: Firebase(Firestore+Auth) 채택 — 서버리스로 GitHub Pages 정적 구조 유지, Spark 무료 플랜으로 충분. 대안 Supabase는 SQL 집계가 편하나 무료 플랜의 1주 미접속 자동 일시정지가 단점
- **Firebase 콘솔 필수 설정**: ① Authentication → Sign-in method에서 **익명** + **이메일/비밀번호** 활성화 ② Firestore 생성(리전 asia-northeast3, 프로덕션 모드) 후 `firestore.rules` 내용을 규칙 탭에 게시 ③ Authentication → Settings → 승인된 도메인에 `woodong84.github.io` 추가 ④ Users에서 관리자 계정 생성. 랭킹이 "불러오지 못했어요"로 실패하면 ①·② 누락이 최우선 의심 대상
- **캐시 문제**: 배포 후 모바일에서 변경이 안 보이면 브라우저 캐시가 원인 — `?v=N` 버전을 올려 배포하면 강제 갱신됨 (기능 변경 없이 화면이 예전 그대로일 때 1순위로 확인)
